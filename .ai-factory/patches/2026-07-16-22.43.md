# Default model settings missing from lazy and eager relations

**Date:** 2026-07-16 22:43
**Files:** src/Relations/SettingsRelation.php
**Severity:** high

## Problem

Lazy loading returned only model-specific settings, so inherited defaults were absent. Eager loading inherited defaults only for parents that already had at least one model-specific setting. A parent without overrides received an empty relation and `get()` returned `null` instead of the default payload.

## Root Cause

The lazy relation fell back to the standard `MorphMany::addConstraints()`, which restricts `item_id` to the parent key and excludes the default sentinel `0`. The eager materialization grouped only model-specific rows and therefore had no group for parent IDs without overrides.

## Solution

Restored `PriorityScope` in `SettingsRelation::addConstraints()` while retaining the morph-type constraint. Cached all eager parent keys before adding the default sentinel to the SQL constraint, then materialized defaults and overrides for every parent key. Inherited defaults remain isolated replicas whose `item_id` matches the target parent, allowing the standard Laravel matcher to attach them correctly.

## Prevention

- Cover lazy and eager reads for a parent with no overrides.
- Verify mixed default and model-specific keys rather than only matching key sets.
- Run relation tests against integer, UUID, and ULID primary keys.
- Preserve the morph-type constraint whenever replacing standard polymorphic relation constraints.
