# Bulk mutation handling coupled the package to application logging

**Date:** 2026-07-19 01:47
**Files:** src/Exceptions/BulkMutationException.php, src/Services/SettingsService.php, tests/Architecture/StrictTest.php, tests/Feature/HasSettings/BulkMutationFailureTest.php, tests/Feature/HasSettings/InvalidSettingKeyTest.php, tests/Feature/HasSettings/SetManyTest.php, tests/Unit/BulkMutationExceptionTest.php, tests/Unit/Casts/StringTest.php, docs/docs/api-reference.md, docs/docs/settings.md, docs/i18n/*/docusaurus-plugin-content-docs/current/api-reference.md, docs/i18n/*/docusaurus-plugin-content-docs/current/settings.md
**Severity:** medium

## Problem

Bulk mutation methods wrote start, preparation, completion, and failure details through Laravel's logging facade. This imposed an observability policy on the host application, while unexpected failures had no stable package-level exception that identified the failed operation and scope.

## Root Cause

`SettingsService` treated logging as part of the package contract. Its bulk mutation catch blocks logged unexpected failures and rethrew their original types instead of exposing a stable package-level failure boundary.

## Solution

Removed the logging facade and every package log call. Unexpected iterable, serialization, and persistence failures from `setMany()`, `forgetMany()`, and `purge()` are now wrapped in `BulkMutationException`. The exception identifies the operation, owner class, and scope without exposing setting keys or payloads, and preserves the original exception through `getPrevious()`.

Existing package exceptions remain unwrapped. Regression tests cover the exception contract, iterable and storage failures, transaction rollback, relation retention, safe messages, and preservation of the original cause. An architecture test prevents production code from depending on Laravel logging.

## Prevention

- Keep application logging policy outside reusable library code.
- Expose stable package exceptions for operation-level failures.
- Preserve original failures through exception chaining.
- Keep exception messages free of setting keys and payloads.
- Enforce the logging boundary with an architecture test.
