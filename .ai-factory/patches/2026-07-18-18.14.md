# Zero-valued primary key fixture failed on MySQL and MariaDB

**Date:** 2026-07-18 18:14
**Files:** tests/Feature/Owner/MutationValidationTest.php, workbench/app/Models/SomeInteger.php, workbench/database/migrations/2026_07_18_000001_create_integer_table.php
**Severity:** medium

## Problem

The MySQL 8 and MariaDB 10 jobs failed four owner-validation scenarios with `ModelNotFoundException` after the tests inserted an owner with primary key `0` and called `findOrFail(0)`.

## Root Cause

The test used the `some_ids` table, whose primary key is auto-incrementing. MySQL and MariaDB interpret an explicit `0` in an `AUTO_INCREMENT` column as a request for the next generated identifier unless `NO_AUTO_VALUE_ON_ZERO` is enabled. The test therefore never persisted the owner under key `0` on those databases.

## Solution

Added a dedicated workbench model and table with a non-auto-incrementing integer primary key. The zero-identifier datasets now use that fixture, while `SomeId` continues to cover normal auto-incrementing identifiers. Strict assertions verify that integer `0` and string `'0'` remain distinct after model hydration.

## Prevention

- Use non-auto-incrementing fixtures when a test requires a persisted integer key of `0`.
- Avoid database-specific SQL modes for cross-database behavior tests.
- Assert hydrated primary-key types when behavior depends on the distinction between integer and string identifiers.
- Run database edge-case tests against every supported database driver.
